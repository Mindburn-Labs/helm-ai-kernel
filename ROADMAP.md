# Roadmap

HELM's public work is constrained to the OSS kernel, CLI, contracts, SDKs, Console, examples, deployment material, and verification artifacts described in `README.md` and `docs/KERNEL_SCOPE.md`.

## Strategic Relationship

Mindburn Labs' conditional strategy may use this Kernel beneath Company Builder
(B2C), Company OS (B2B), and HELM Network experiences. That does not add their
commercial journeys, revenue targets, plan codes, Network writes, or customer
claims to the OSS roadmap. The Kernel's role remains the open, model-neutral,
fail-closed authority and proof boundary. External builders and agents may
route actions through it; their planning and generation do not move into Kernel
authority semantics.

## Now

- Make the GitHub landing page convert first-time visitors into stars, discussions, local demo runs, and first issues.
- Keep the no-signup local proof path visible: MCP quarantine, OpenAI-compatible proxy receipts, and tamper-failed receipt verification.
- Maintain a newcomer backlog across docs, examples, MCP fixtures, proxy examples, receipt verification, SDK polish, and ecosystem mapping.
- Keep public docs and package install instructions tied to verified release state.

## Next

- Expand runnable integration guides for MCP servers, AI gateways, security scanners, guardrails, and agent frameworks.
- Add more source-backed launch assets: terminal recordings, proof screenshots, and short posts that explain the execution-firewall model.
- Grow the public maintainer set beyond the current Mindburn Labs core team.
- Turn recurring discussion requests into labeled issues and small PRs.

## Later

- Add production hardware validation before describing vendor TEE attestation as ready for production deployments.
- Broaden conformance fixtures for ecosystem integrations once local examples are stable.
- Keep hosted Mindburn Labs product claims separate from OSS kernel claims unless source-backed evidence exists in this repository.

Release-level changes continue to be documented in `CHANGELOG.md`; release security posture is documented in `docs/RELEASE_SECURITY.md`.
